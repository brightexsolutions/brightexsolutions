Data Access Rules

Golden Rule
The service_role Supabase key never leaves the server. All data reads/writes happen in API Route Handlers or Server Components. Client Components call /api/* routes — they never touch Supabase directly. Supabase RLS blocks everything from the anon key as a hard second line of defence.

Middleware

src/middleware.ts intercepts all /admin/*, /work/*, and /team/* routes. It checks the Supabase session cookie and redirects unauthenticated requests to the appropriate login page. For team routes it also verifies the app_metadata.app_role JWT claim matches the expected role.

Admin Impersonation — "Login as Member"

From /admin/team the admin can open any portal-member's account in a new browser tab for quick diagnosis or testing.

Step 1
Admin clicks "Login as member" in the team member's row or detail panel
Step 2
POST /api/admin/team/[id]/impersonate — server calls Supabase Admin generateLink({ type: 'magiclink', email }), extracts the raw OTP token from the returned action_link URL, and returns { token, email, next }
Step 3
Browser opens /auth/impersonate?token=…&email=…&next=/team/finance in a new tab (outside the Supabase redirect URL whitelist — no dashboard config needed)
Step 4
/auth/impersonate calls supabase.auth.verifyOtp({ email, token, type: 'magiclink' }) client-side — exchanges the token for a live session
Step 5
On success, hard-navigates to next (e.g. /team/finance). The tab is now signed in as the team member. The admin's original session is unaffected in the original tab.
Important
The impersonation token is single-use. The original admin session is NOT disturbed — this opens a separate session in the new tab only. The action is recorded in the Activity Log under entity type team, action impersonated.

External Contractors (No Portal)

External/outsourced subcontractors are added with the "External Only" toggle. They receive:

  • A team_members record with user_id = null (no Supabase Auth account)
  • Full visibility in the admin — can be assigned tasks, have payouts recorded, have expenses logged
  • No portal access — they never log in; you track work on their behalf

Workflow
Projects & Collaboration

Assigning a Subcontractor to a Project

  1. Open /admin/projects → click a project to open the detail sheet
  2. Scroll to the Tasks section → click Add Task
  3. Fill in title, priority, due date, and select a subcontractor in the Assign To dropdown
  4. The dropdown lists all active subcontractors (internal with portal + external with no portal)
  5. The task now appears in the subcontractor's /work portal (if they have portal access)
  6. External contractors are assigned the same way — their tasks are visible in admin but they don't see a portal

Task Lifecycle

todo → in_progress → review → done

  • Subcontractor can advance their own task status from their portal
  • Admin can advance any task status from the project detail sheet
  • Subcontractor can upload a deliverable URL at any status
  • Every status change and deliverable upload is logged to the activity log
  • Progress notes are created as a "done" task immediately — they appear in the task list for the project

finance_documents SQL

CREATE TABLE finance_documents (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  doc_type         text NOT NULL CHECK (doc_type IN ('etims_invoice','supplier_invoice','client_receipt','expense_receipt','other')),
  direction        text NOT NULL CHECK (direction IN ('income','expense')),
  party_name       text NOT NULL,
  invoice_number   text,
  amount           numeric,
  currency         text DEFAULT 'KES',
  doc_date         date NOT NULL,
  storage_path     text NOT NULL,
  original_filename text,
  size_bytes       integer,
  linked_income_id  uuid REFERENCES income_records(id) ON DELETE SET NULL,
  linked_expense_id uuid REFERENCES expenses(id) ON DELETE SET NULL,
  notes            text,
  uploaded_by      uuid REFERENCES auth.users(id),
  created_at       timestamptz DEFAULT now()
);
GRANT SELECT, INSERT, UPDATE, DELETE ON finance_documents TO service_role;
Supabase Storage Setup
Create a private bucket named finance-docs in your Supabase project (Storage → New bucket → Private). Files are accessed exclusively through 1-hour signed URLs generated server-side. The finance-docs bucket stores files under the path {direction}/{uuid}.{ext} (e.g. income/abc123.pdf).
Supabase GRANT Note
All new Supabase projects (May 2026+) require explicit GRANT SELECT, INSERT, UPDATE, DELETE ON table_name TO service_role; after each CREATE TABLE. Include these in all migration SQL files.

Automation
Cron Jobs Reference
How Crons Work
We do NOT use Vercel's built-in cron (limited to 2 jobs on Hobby, daily minimum). Instead, cron-job.org (free, unlimited jobs, 1-minute minimum) sends a GET request to each API route on schedule. Every cron route verifies x-cron-secret: CRON_SECRET header — requests without it return 401.

Setup on cron-job.org

  1. Create a free account at cron-job.org
  2. For each job below: click "Create Cronjob" → set URL, schedule, and add the custom request header x-cron-secret: <your CRON_SECRET value>
  3. Enable email alerts on 3 consecutive failures for all jobs
  4. Test each job manually by clicking "Execute now" — confirm it returns 200 with a JSON body

Job Schedule

RouteSchedulePurpose
/api/healthEvery 5 minutesPublic uptime check — no secret needed. cron-job.org alerts on 3 consecutive non-200 responses.
/api/cron/db-keepaliveEvery 12 hoursPrevents Supabase free tier DB pause (pauses after 7 days inactivity). Runs a lightweight SELECT 1.
/api/cron/site-healthEvery 6 hoursPings all registered sites, records status/response time/SSL expiry, checks WP updates, generates system_alerts.
/api/cron/payment-remindersDaily — 08:00 EATEmails overdue invoice reminders to clients. Skips already-reminded invoices. Logs to activity_log.
/api/cron/booking-remindersDaily — 07:00 EATSends 24h-ahead reminder emails for confirmed bookings. Sets reminder_sent = true to prevent duplicates.
/api/cron/subscription-alertsDaily — 09:00 EATFlags subscriptions renewing within 14 days as system_alerts (warning) or past renewal date (critical).
/api/cron/trial-expiryDaily — 10:00 EATMarks product trials past 7 days as expired, sends "subscribe to continue" email to each trial requester.
/api/cron/cleanupSundays — 02:00 EATArchives old replied contacts (>90 days), removes stale draft invoices (>60 days), prunes acknowledged alerts.
Critical: DB Keepalive Must Always Run
If the db-keepalive cron stops firing for >7 days, the Supabase free tier database will pause. Recovery requires logging into the Supabase dashboard and manually unpausing. Set up the cron-job.org email alert to catch this before it happens.

Deployment
Netlify → Vercel Migration Guide
Expected Downtime
Zero visible downtime when done in the correct order. Vercel builds and serves the new version before DNS is switched. DNS propagation varies (15 min to 48h) — during this period some visitors may still hit the old Netlify site, which is fine since it remains live until you remove the domain there.

Phase 1 — Deploy to Vercel (before touching DNS)

  • 1 Create a Vercel account at vercel.com and import the GitHub repository
  • 2 Set the Root Directory to brightex-website (or wherever package.json lives)
  • 3 Set Framework Preset to Next.js
  • 4 Add all environment variables in Vercel → Project → Settings → Environment Variables (copy from your local .env.local)
  • 5 Deploy — Vercel will give you a preview URL like brightex-website.vercel.app

  • 1 Log into your domain registrar (wherever you registered brightexsolutions.co.ke)

  • 3 For the apex (@ / root): change the A record to Vercel's IP (shown in your Vercel domain settings, typically 76.76.19.61)

Future Roadmap — Post-Launch Opportunities

These are ideas to pursue after the platform is stable in production. They are organised by impact and effort.

High Impact, Low Effort

  • Client portal — Let clients log in to view their project status, invoices, and download receipts. Same pattern as team portals — new role client with read-only access to their own data.
  • WhatsApp Business API integration — Replace the wa.me link with actual WhatsApp message delivery for payment reminders and booking confirmations. Meta Business API + Twilio or 360dialog (both have free tiers for low volume).
  • Mpesa Daraja integration — Receive M-Pesa STK push payments directly from invoice links. Auto-marks invoice paid and records payment. Daraja API is free (Safaricom paybill/till required).
  • Email auto-replies for enquiries — When a contact form submission comes in, Brixo can draft a personalised reply based on the service enquired about, for Godwin to review and send with one click.

Medium Impact, Medium Effort

  • Proposal builder — Generate PDF proposals from a template, pre-filled with client name, service, scope, and pricing. Attach to the sales pipeline and send directly from the admin.
  • Time tracking — Allow subcontractors to log hours per task. Auto-calculate payout amounts for hourly-rate contractors. Integrates with the existing task and payout tables.
  • Recurring invoice automation — Retainer invoices are already drafted monthly — automate the send (Godwin still approves, but clicks once instead of four times).
  • Product subscription billing — Automated renewal reminders + payment link for product subscribers. Currently reminders are manual.
  • Brixo AI upgrade — Replace keyword FAQ matching with an LLM-based response (Claude API) that answers naturally. The current session storage and escalation flow remains the same — only the matching logic changes.
  • Google Calendar sync — Sync admin calendar events to Google Calendar via Google Calendar API. Two-way or one-way. Free API quota for personal use.

High Effort, High Long-term Value

  • Multi-tenant SaaS — The platform architecture (Next.js + Supabase + role-based access) is a solid foundation for a productised version sold to other agencies. Each agency gets their own Supabase project (or a schema-per-tenant approach).
  • Mobile app (React Native) — A lightweight companion app for Godwin: push notifications for new bookings, quick invoice approval, task status updates. Expo + Supabase is a natural stack extension.
  • Analytics dashboard — Business intelligence layer on top of existing data: revenue trends, client acquisition sources, project profitability, team productivity metrics. Recharts is already installed.
  • Automated social media publishing — Replace the current "mark as posted manually" flow with actual API publishing to Instagram, Facebook, LinkedIn via their respective APIs. Requires business account verification on each platform.
  • Contract management — Digital contract templates, e-signature via Docusign or PandaDoc API (both have free tiers), auto-linkage to projects and clients.
  • Supplier/vendor management — Track recurring vendors (hosting providers, freelancers, tool subscriptions) beyond the current expense categories. Purchase orders, vendor contacts, payment history per vendor.

Infrastructure Upgrades to Watch For

  • When Gmail hitting daily limits → switch to Resend (3K free emails/month, Nodemailer-compatible, change 3 config lines)
  • When Supabase DB approaches 400MB → audit large tables, add indexes, consider archiving old activity_log entries
  • When deploying across multiple regions → replace in-memory rate limiter with Upstash Redis (10K commands/day free) for shared rate limit state
  • When Vercel bandwidth approaches 80GB/month → upgrade to Pro for compliance + team features
